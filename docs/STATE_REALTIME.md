# Состояние, Realtime и Optimistic Updates

Этот документ описывает “сердце” приложения — `usePlanner`.

## Источник истины

- `tasks: Task[]` — задачи в активном месяце (по `selectedDate`)
- `goals: Task[]` — цели по активным периодам (day/week/month/year)
- derived state:
  - `currentTasks` (фильтр по selectedDate)
  - `taskDates` (Set дат с задачами)
  - `goalsByPeriod` (группировка + сортировка по slot)
  - `hours/minutes` (остаток времени по незавершенным)
  - `active task timer` — `elapsed_ms` + `active_started_at` в `tasks`

## Ручной порядок (drag & drop)

Порядок таскается только там, где это действительно важно:

- группы приоритета остаются главными: активные → закреплённые → обычные → выполненные
- drag‑drop применяется ко всем **незавершённым и неактивным** задачам внутри своей группы
- `start_minutes` не влияет на ручной порядок (это время начала и напоминаний)

Это делает порядок предсказуемым и не ломает логику групп.

## Паттерн optimistic insert (почему так)

Когда пользователь добавляет задачу/цель:

1. создаём `tempId`
2. сразу добавляем в UI
3. пишем в БД
4. когда приходит INSERT (realtime) или ответ INSERT:
   - сопоставляем запись и заменяем tempId на настоящий id
   - предотвращаем дубликаты

### Как сопоставляем temp insert с реальным INSERT

В коде есть `pendingTaskInsertRef` / `pendingGoalInsertRef` (Map tempId → row поля).
Когда приходит INSERT:

- ищем совпадение по ключевым полям (title, duration, date, completed, position, color, is_pinned, checklist; для goal ещё period/slot)
- если нашли: заменяем запись tempId на реальную и удаляем pending
- если не нашли: добавляем как новую (например добавлено с другого устройства)

⚠️ Правило: при расширении модели задач дополняйте критерий сопоставления, иначе появятся дубликаты. Для повторов в сопоставление добавлен `series_id`.

## Паттерн optimistic toggle (гонки)

toggleTask/toggleGoal используют:

- `toggleRequestRef` (Map id → requestId)
- при каждом toggle увеличиваем requestId, и только последний ответ считается актуальным

Это защищает от:

- двойных кликов
- задержек сети
- out-of-order ответов

## Таймер задачи (sync между устройствами)

- Источник истины: поля `tasks.elapsed_ms` и `tasks.active_started_at`.
- Включение/выключение — через RPC `toggle_task_timer(task_id)`, который:
  - останавливает предыдущую активную задачу пользователя,
  - запускает новую (если она не выполнена).
- Завершение задачи автоматически останавливает таймер на уровне БД (trigger).

## Delete + Undo

Delete делается optimistic:

- удаляем из state
- если БД ответила ошибкой — вставляем обратно по прежнему индексу
  Undo (restoreTask) — это новый INSERT.

Best practice:

- для undo-операций держать срок жизни (TTL) UI-сообщения и чистить таймеры на unmount.

## Realtime подписка: правила

Подписка фильтруется по `telegram_id=eq.${userId}` и table `tasks`.
На приходящих событиях:

- INSERT:
  - goal: добавлять только если goal в активном периоде
  - task: добавлять только если date в активном месяце
- UPDATE:
  - если запись “выпала” из активного диапазона — удалить из state
- DELETE:
  - аналогично — удаляем только если попадает в активный диапазон

Best practice:

- держать фильтрацию “на входе” чтобы state не разрастался.
- всегда защищаться от `row?.id` отсутствия.

## Ошибки сети и offline

Рекомендуемое поведение:

- Optimistic UI остаётся.
- Если операция не прошла — откат + haptic “error”.
- Для будущего: можно добавить очередь offline операций, но это отдельная фича.

## Рекомендации по рефакторингу (если станет сложно)

Если `usePlanner` разрастётся:

- вынести Supabase операции в `app/services/tasks.ts`
- оставить в хуке только orchestration и state
- сделать отдельные хуки `useTasksMonth`, `useGoalsPeriods`
