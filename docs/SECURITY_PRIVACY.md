# Security & Privacy

## Модель угроз (простая)

Атакующий может:

- подделывать запросы к `/api/auth/telegram`
- пытаться читать/писать чужие данные в Supabase
- пытаться украсть токен/секреты

## Главные защиты

### 1) Проверка Telegram initData

- timing-safe сравнение hash
- `auth_date` ограничен по времени
- bot token хранится только на сервере

### 2) Supabase RLS по claim `telegram_id`

- даже если кто-то получил anon key, без валидного JWT он не увидит чужие строки
- `telegram_id` в строках привязан к JWT по default и policy

### 3) Разделение env

- никаких секретов в `NEXT_PUBLIC_*`

## Практики безопасной разработки

- Не логировать `initData`, JWT и персональные данные.
- Обрабатывать ошибки одинаково “неразговорчиво” (особенно на auth).
- TTL токена держать относительно коротким (сейчас 24h — разумно для мини-аппа).
- При добавлении новых таблиц:
  - сразу включать RLS
  - писать политики “только своё”
  - добавлять ограничения целостности

## Приватность

- Мы храним минимальный идентификатор: `telegram_id`.
- Не добавляйте `username/first_name/last_name` в БД без явного смысла и политики хранения.
- Если добавите аналитику — убедитесь, что она не собирает PII и соответствует требованиям юрисдикций.
