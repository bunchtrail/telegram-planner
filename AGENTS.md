# AGENTS.md — правила работы ИИ-агента в этом репозитории

Этот репозиторий — Telegram Mini App (планер) на Next.js (App Router) + Supabase (RLS + Realtime).
Цель этого файла: дать агенту ясные правила, чтобы изменения были безопасными, типобезопасными и согласованными.

## Что обязательно прочитать перед изменениями

1. `docs/ARCHITECTURE.md` — общая архитектура и границы ответственности
2. `docs/AUTH_TELEGRAM.md` — Telegram initData, валидация, JWT
3. `docs/DATA_MODEL_SUPABASE.md` — таблица `tasks`, RLS, ограничения, индексы
4. `docs/STATE_REALTIME.md` — оптимистичные обновления и reconciliation
5. `docs/FRONTEND_GUIDE.md` + `docs/DESIGN_SYSTEM.md` — UI/UX, a11y, анимации, токены

## Определение “готово” (Definition of Done)

Изменение считается готовым, если:

- ✅ TypeScript strict не нарушен (нет `any` без причины, нет сломанной типизации).
- ✅ UX в Telegram Mini App не деградировал (скролл, safe-area, клавиатура, haptics).
- ✅ Безопасность не ухудшилась (нет утечек секретов, RLS остаётся корректным).
- ✅ Нет регрессий в realtime/optimistic логике (нет дубликатов, гонок, несогласованностей).
- ✅ Добавлены/обновлены docs при изменении архитектуры/контрактов/API/схемы.
- ✅ `npm run lint` и `npm run build` проходят локально.

## Нельзя делать (hard rules)

- ❌ Нельзя переносить секреты в `NEXT_PUBLIC_*`.
- ❌ Нельзя отключать RLS или ослаблять политики без явного описания в docs и миграции.
- ❌ Нельзя делать запросы к Supabase от имени другого `telegram_id` (всё держится на claim `telegram_id`).
- ❌ Нельзя ломать совместимость с Telegram WebApp окружением (initData, haptics, safe-area).
- ❌ Нельзя “чинить” гонки увеличением задержек/таймаутов без анализа причин.

## Принципы проектирования

- Минимизировать “ум” в компонентах — бизнес-логика в хуках/сервисах.
- Изолировать внешние интеграции: Telegram API — в `app/hooks/useHaptic.ts` и маленьких утилитах; Supabase — в `app/lib/supabase.ts` и/или сервисных модулях.
- Стабильные контракты: типы данных — в `app/types/`, константы — в `app/lib/constants.ts`.
- Обновления схемы БД только через миграции/SQL и синхронизацию docs.

## Шаблон работы агента (рекомендуемый)

1. Сформулировать цель и ограничения (UX/безопасность/данные).
2. Найти точку расширения (компонент / хук / API route / схема).
3. Сделать минимальную реализацию + типы + обработку ошибок.
4. Проверить: optimistic/realtime, a11y, mobile/safe-area, perf.
5. Обновить документацию (если поменялись контракты или подход).
6. Прогнать линт/билд.

## Чеклист для PR (или для финальной проверки)

- [ ] Нет изменений в секретах/ENV, которые экспонируются в браузер
- [ ] API-ответы не кэшируются там, где нельзя (auth — `Cache-Control: no-store`)
- [ ] Любая новая таблица/колонка — с RLS, индексами и ограничениями целостности
- [ ] UI не ломает клавиатуру (iOS), safe-area, scroll-lock
- [ ] Анимации уважают `prefers-reduced-motion`
- [ ] Даты хранятся и интерпретируются последовательно (см. docs)
